## Ingest — 消化新知识

### Ingest：论文

```
请把以下论文整理成知识库条目。

[粘贴论文标题、摘要、或 arXiv 链接]

要求：
1. 先判断类型：这篇论文主要贡献是提出新概念/方法（→ concepts），还是介绍一个模型/系统（→ entities）？
   同时创建一个 sources 摘要页。
2. concepts 或 entities 页：重点讲清楚原理、核心公式（LaTeX）、与现有条目的关系
3. sources 页：记录论文基本信息、核心贡献3条、关键公式、延伸阅读
4. 在相关的现有条目的 related 字段里加上新条目的 slug
5. 更新 _wiki/log.md，追加一条 ingest 记录
```

---

### Ingest：博客文章

```
请把这篇博客文章提炼成知识库条目：

[粘贴文章标题或文件名，如 _posts/2025-02-17-一文搞懂KVCache.md]

要求：
1. 读取博客原文，提取核心知识点
2. 创建或更新对应的 concepts/entities 条目（如已存在则补充内容）
3. 在条目底部的"参考资料"中注明来自博客并附链接
4. 不要修改原始博客文章
5. 更新 log.md
```

---

### Ingest：概念速记

当你想快速记下一个概念但没有完整资料时：

```
帮我在知识库里创建一个 [概念名] 的条目。

我知道的信息：
[用自己的话描述，哪怕不完整]

要求：
- 基于我提供的信息写初稿
- 标注哪些地方需要后续补充（用 > ⚠️ 待补充：xxx 格式）
- description 字段先写一行简介
- 更新 log.md
```

---

### Ingest：批量处理

```
请扫描 _posts/ 目录下所有博客文章，找出哪些文章涉及的核心概念在 _wiki/ 中还没有对应条目，
列出清单（标题 + 建议的 slug + 建议的 wiki_type），然后等我确认后再逐个处理。
```

## Query — 知识问答

### Query：知识问答

```
根据知识库回答以下问题，优先阅读 _wiki/ 目录，再结合 _posts/ 博客：

[你的问题]

回答格式：
1. 直接给出答案
2. 标注引用了哪些 wiki 条目和博客文章
3. 如果答案值得保存，帮我把它存为新的 wiki 条目
```

---

### Query：概念对比

```
请对比以下两个/多个概念，从知识库和博客中整理信息后，生成一个对比表格：

概念列表：[概念A、概念B、概念C]

对比维度：[如：原理、计算复杂度、适用场景、代表模型] （不确定就让 Claude 自己选）

输出后问我：是否要把这个对比表保存为 wiki 条目？
```

---

### Query：概念地图

```
以 [某个核心概念，如 "注意力机制"] 为中心，请：
1. 列出 _wiki/ 中所有与它直接相关的条目
2. 列出 _posts/ 中涉及它的博客文章
3. 找出"应该有但还没有"的周边条目（stub列表）
4. 画出文字版关系图（用缩进表示层级）
```

---

### Query：论文解读

```
帮我解读这篇论文，结合知识库已有内容，重点解释对我来说有哪些新增知识：

[论文标题/摘要]

已有背景（Claude 会自动读取知识库，不用你说）。
重点告诉我：
1. 这篇论文解决了什么问题？
2. 核心方法/公式是什么？
3. 与我知识库里哪些条目相关？
4. 有哪些值得深入的点？
```

## Blog — 博客写作

### Blog：新建文章

```
帮我起草一篇博客文章，主题是：[主题]

要求：
1. 文件名格式：_posts/YYYY-MM-DD-标题.md（使用今天日期）
2. Front matter：layout: post，categories 和 tags 参考现有博客风格
3. 风格参考我已有的博客（可以读几篇 _posts/ 下的文章感受风格）
4. 包含必要的 LaTeX 公式
5. 大纲先给我确认，再写正文
```

---

### Blog：博客 → Wiki 同步

```
我刚写完一篇博客：[文件名]

请：
1. 检查这篇文章涉及的核心概念，是否在知识库中已有条目
2. 没有的帮我创建简要条目，并在条目中链接回博客
3. 已有的检查内容是否需要更新
4. 给我一个同步报告
```

## Lint — 知识库维护

### Lint：健康检查

```
对知识库做一次完整健康检查，检查项：

1. **孤立页面**：related 字段引用了不存在的 slug
2. **单向链接**：A 引用了 B，但 B 没有引用 A（双向不完整）
3. **空 description**：description 字段为空或过长（>80字）
4. **Stub 候选**：多个页面提到某个概念但无独立条目
5. **过期内容**：updated 日期超过6个月的条目，检查是否需要更新
6. **log.md 待补充**：log.md 里 [ ] 未完成的 stub，优先级排序

输出：问题清单 + 建议修复顺序
```

---

### Lint：填充 Stub

```
查看 _wiki/log.md 里"待补充条目"列表，
选择其中 [N 个/某个主题方向] 来创建条目。

优先选择：已有博客文章作为来源的条目（可以从博客提炼）。
```

---

### Lint：索引重建

```
重新生成 _wiki/index.md 的内容：
1. 统计当前各类型条目数量
2. 更新"最近更新"列表
3. 更新"待补充条目"列表（从各条目的 related 中找到未创建的 slug）
```

## Bootstrap — 从零开始一个新主题

当你想系统学习一个新方向时：

```
我想在知识库里系统整理关于 [主题，如"模型量化"/"MoE"] 的知识。

请：
1. 列出这个主题下应该有哪些核心条目（5-10个，含建议 slug 和类型）
2. 按"先学什么后学什么"的顺序排列建议阅读顺序
3. 我现有博客中哪些文章可以作为来源？
4. 还需要补充哪些外部资料？

列完后等我确认，再开始逐个创建。
```

## 高效使用技巧

### 技巧 1：Obsidian + Claude Code 并排工作

左边 Obsidian 查看 wiki 图谱，右边 Claude Code 终端。  
让 Claude 修改文件，Obsidian 实时预览结果。

### 技巧 2：让好的问答沉淀为条目

每次获得有价值的解释后加一句：
```
把这个解释整理成一个 wiki 条目保存
```

### 技巧 3：渐进式 Ingest

不需要一次读完整篇论文。可以分步：
```
第一步：先创建一个 stub 条目，只填 description 和 tags
第二步：[读完论文后] 补充核心内容和公式
第三步：添加 related 链接
```

### 技巧 4：利用搜索而非背记

wiki 的价值不是"记住"而是"快速找到"。  
写条目时重点保留：难以推导的公式、各方法的对比表、踩过的坑。

### 技巧 5：周度 Lint

每周花 5 分钟跑一次 Lint，让知识库保持健康：
```
快速 lint：只检查孤立 slug 和空 description
```

## Claude Code 行为约定

Claude Code 在操作本仓库时应遵守：

1. **不修改** `_posts/` 下已有文章
2. **不修改** `css/site.css`、`css/blog.css`（除非专门讨论样式）
3. **每次 ingest 必须** 更新 `_wiki/log.md`
4. **新建条目前** 先检查 `_wiki/` 是否已有相似条目（避免重复）
5. **related 字段** 只填已存在的 slug，未存在的在 log.md 中标注为 stub
6. **中文为主**，专有名词（模型名、论文标题、作者名）保留英文原文
7. **公式** 用 LaTeX（行内 `$...$`，块级 `$$...$$`）
8. **description** 字段控制在 80 字符以内

完整操作规范见 [`CLAUDE.md`](./CLAUDE.md)。
